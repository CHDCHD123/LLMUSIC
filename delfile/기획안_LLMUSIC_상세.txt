프로젝트 기획안: LLMUSIC - 감정 기반 음악 추천 및 차트 분석 시스템

1. 프로젝트 개요

LLMUSIC 프로젝트는 지니뮤직 차트 데이터 크롤링부터 LLM(Large Language Model) 기반의 감성 음악 추천 및 차트 변화 분석 보고서 생성까지, 음악 데이터와 AI 기술을 통합한 엔드투엔드 시스템 구축을 목표로 합니다. 사용자의 감정과 상황에 맞는 음악 추천 서비스를 제공하고, 음악 차트의 동향을 심층적으로 분석하여 인사이트를 도출합니다.

2. 프로젝트 배경 및 필요성

2.1. 기존 음악 추천 서비스의 한계
현재 대부분의 음악 추천 서비스는 사용자 청취 이력, 장르 선호도 등 정량적 데이터에 기반하고 있습니다. 이는 사용자의 순간적인 '감정'이나 '상황'을 반영하지 못하여, 때때로 사용자의 현재 니즈와 동떨어진 추천을 제공하는 한계가 있습니다. 사용자는 단순히 좋아하는 장르를 넘어, 지금 느끼는 감정에 깊이 공감하고 어울리는 음악을 원합니다.

2.2. 개인화된 음악 경험의 중요성 증대
개인화된 콘텐츠에 대한 수요가 증가하면서, 음악 분야에서도 사용자 개개인의 미묘한 감정 변화와 특정 상황에 최적화된 추천이 중요해지고 있습니다. 이는 사용자 만족도와 서비스 충성도 향상에 직결되는 핵심 요소입니다.

2.3. 빠르게 변화하는 음악 차트 동향 분석의 필요성
음악 차트는 대중의 음악 취향과 사회적 트렌드를 반영하는 중요한 지표입니다. 그러나 방대한 데이터를 수동으로 분석하는 것은 비효율적이며, 핵심적인 인사이트를 적시에 도출하기 어렵습니다. AI를 활용한 자동화된 차트 분석은 의사 결정에 필요한 심층적인 정보를 신속하게 제공할 수 있습니다.

2.4. LLM 기술의 음악 서비스 적용 가능성 탐색
최근 발전하고 있는 LLM 기술은 자연어 이해 및 생성 능력을 통해 복잡한 데이터를 해석하고, 인간과 유사한 방식으로 소통하며 설명을 제공할 수 있습니다. 이를 음악 추천 및 차트 분석 영역에 적용하여 기존 서비스의 한계를 극복하고 새로운 가치를 창출할 수 있는 가능성을 탐색합니다.

3. 프로젝트 목표

3.1. 자동화된 음악 차트 데이터 파이프라인 구축
지니뮤직 TOP 200 차트 데이터를 정기적으로 수집하고, 일간 차트 변화를 자동으로 분석하는 시스템을 Apache Airflow 기반으로 구현합니다.

3.2. LLM 기반 감정 음악 추천 서비스 개발
사용자 감정과 상황에 맞춰 Spotify, Last.fm, 지니뮤직 차트 데이터를 활용한 개인화된 음악 추천 서비스를 제공하고, LLM을 통해 추천 이유를 자연어 설명하여 사용자 공감대를 형성합니다.

3.3. AI 기반 차트 분석 보고서 생성
일간 차트 변화 데이터를 바탕으로 LLM이 핵심 동향을 파악하고 보고서를 자동 생성하여, 음악 산업 관계자 및 일반 사용자에게 시의성 있는 데이터 기반 인사이트를 제공합니다.

3.4. 사용자 친화적인 인터페이스 제공
Streamlit 앱 및 Flask 기반 웹 애플리케이션을 통해 직관적이고 반응성 있는 사용자 경험을 제공하며, 시스템의 접근성을 높입니다.

4. 서비스 대상

*   **음악 추천이 필요한 일반 사용자:** 자신의 감정과 상황에 맞는 새로운 음악을 탐색하고 싶은 사용자.
*   **음악 산업 관계자:** 빠르게 변화하는 음악 차트 동향을 분석하고, 시의성 있는 보고서를 통해 마케팅 및 기획 전략을 수립하고자 하는 전문가.
*   **데이터 분석가/연구자:** LLM 기반의 데이터 해석 및 보고서 자동화 기술에 관심 있는 전문가.

5. 주요 기능 및 상세

5.1. 음악 차트 데이터 크롤링 (Crawling Genie Chart)
*   **담당 DAG:** `dags/crawler_genie.py`
*   **주요 기능:**
    *   지니뮤직 TOP 200 차트 페이지에서 곡명, 아티스트, 순위, 앨범명, 장르, 재생시간, 좋아요 수, 전체 청취자 수, 전체 재생 수, 곡 ID, 상세 URL 등 모든 공개 정보 자동 수집.
    *   User-Agent 및 Referer 헤더 설정, `time.sleep`을 통한 요청 간격 조절로 웹 크롤링 차단 회피 로직 포함.
    *   크롤링된 데이터는 일별로 `data/genie_top100_YYYY-MM-DD.csv` 형태로 저장 (UTF-8 BOM 인코딩).
*   **기술:** Python, `requests`, `BeautifulSoup4`, `pandas`, `datetime`, `os`
*   **사용자 시나리오:** Airflow 스케줄러가 매일 자정 또는 특정 시간에 이 DAG를 실행하여 최신 지니뮤직 차트 데이터를 자동으로 확보합니다.

5.2. 차트 변화 분석 (Analyze Chart Diff)
*   **담당 DAG:** `dags/diff_genie.py`
*   **주요 기능:**
    *   `data/` 폴더 내 최신 2일간의 `genie_top100_*.csv` 파일을 로드하여 비교.
    *   **차트 변동 분류:** 곡 ID를 기준으로 '신규 진입', '차트 이탈', '유지' 곡을 분류.
    *   **변화량 및 변화율 계산:** '유지' 곡에 대해 순위 변동 (상승/하락), 좋아요 수, 전체 청취자 수, 전체 재생 수의 변화량 및 변화율(%)을 계산.
    *   **장르 변화 요약:** 장르별 차트 내 곡 수 변화량을 집계하여 트렌드 파악.
    *   **데이터 정제:** CSV 로드 시 누락된 컬럼 검증, 숫자형 데이터 안전 변환.
    *   **출력 파일:**
        *   상세 비교 결과: `data/genie_diff_YYYY-MM-DD.csv` (UTF-8 BOM 인코딩)
        *   LLM 보고서 생성을 위한 요약 JSON: `data/genie_diff_brief_YYYY-MM-DD.json` (주요 변화 및 TOP N 리스트 포함, LLM 프롬프트에 최적화된 구조)
*   **기술:** Python, `pandas`, `numpy`, `glob`, `os`, `datetime`, `json`
*   **사용자 시나리오:** 크롤링 DAG 완료 후, 이 DAG가 실행되어 차트 변화의 핵심 데이터를 자동으로 추출하고 보고서 생성을 위한 중간 데이터를 준비합니다.

5.3. LLM 기반 차트 분석 보고서 생성 (Generate LLM Report)
*   **담당 DAG:** `dags/jsontxt_genie.py`
*   **주요 기능:**
    *   `data/genie_diff_brief_YYYY-MM-DD.json` 파일을 읽어 LLM 입력 프롬프트로 구성.
    *   Google Gemini API (`gemini-2.0-flash` 모델)를 호출하여 JSON 데이터를 기반으로 한 자연어 차트 분석 보고서 생성.
    *   보고서에는 '신규 진입', '차트 이탈', '주요 순위 변동', '좋아요/청취자 수/재생 수 변화율이 큰 곡', '장르별 변화' 등의 내용이 포함됨.
    *   보고서는 `data/genie_report_YYYY-MM-DD.txt` 파일로 저장 (TSV 형태로, 각 줄이 탭으로 구분).
*   **기술:** Python, `google-generativeai`, `json`, `os`, `datetime`, `pandas`
*   **사용자 시나리오:** 차트 변화 분석 DAG 완료 후, 이 DAG가 실행되어 일간 차트 동향에 대한 전문적인 분석 보고서를 자동으로 생성합니다. 이 보고서는 음악 산업 관계자들이 빠르게 시장 트렌드를 파악하는 데 활용될 수 있습니다.

5.4. 감정 기반 음악 추천 서비스 (Emotion-based Music Recommendation)
*   **담당 모듈:** `MusicRecommender` 클래스 (Shared logic)
*   **UI 인터페이스:**
    *   **`app.py` (Streamlit 앱):** 초기 프로토타입 및 대화형 대시보드.
    *   **`server.py` (Flask API 서버) + `index.html` / `js/main.js` / `css/main.css` (웹 프론트엔드):** 최종 웹 서비스 형태.
*   **주요 기능:**
    *   **사용자 감정 및 상황 입력:**
        *   Streamlit: 감정은 버튼형 카드 선택, 상황은 텍스트 입력.
        *   웹 프론트엔드: 감정은 버튼형 카드 선택, 상황은 텍스트 입력, 한국어 전용 추천 체크박스.
    *   **음악 추천 엔진:**
        *   **Spotify API 활용:** 사용자 입력 감정/상황 키워드를 기반으로 Spotify에서 음악 검색. 한국어 전용 옵션 선택 시 특정 한국 아티스트 목록에서 우선 검색.
        *   **Last.fm API 활용:** 태그 기반으로 추가적인 음악 추천 제공 (주로 장르/분위기 태그).
        *   **지니뮤직 차트 백업:** `data/genie_diff_brief_YYYY-MM-DD.json` 파일에서 최신 지니뮤직 차트 데이터를 활용하여 감정별 선호 장르에 맞는 곡을 백업 추천.
    *   **멀티 LLM 기반 추천 이유 설명:**
        *   사용자 입력과 추천된 곡 목록을 LLM에 전달하여, 해당 곡들이 왜 감정/상황에 어울리는지 2-3문장으로 자연스럽게 한국어로 설명.
        *   **LLM 연동 전략:**
            *   **Google Gemini (gemini-2.0-flash):** 우선적으로 사용. 높은 성능과 유연성을 바탕으로 추천 이유를 설명.
            *   **Qwen/Qwen2.5-1.5B-Instruct (Hugging Face Transformers):** Gemini 연결 실패 시 대체 모델로 활용. 로컬 또는 사내 환경에서 구축 가능한 경량 모델.
            *   **Microsoft/DialoGPT-medium (Hugging Face Transformers):** Qwen도 실패 시 최종 대체 모델. 간단한 설명 생성에 활용.
    *   **서비스 연결 상태 표시:** Spotify, Last.fm, LLM의 API 연결 상태를 실시간으로 표시하여 서비스 신뢰도를 제공.
*   **기술:**
    *   Python, `streamlit`, `flask`, `flask-cors`, `requests`, `spotipy`, `google-generativeai`, `transformers`, `dotenv`
    *   HTML, CSS, JavaScript (웹 프론트엔드)
*   **사용자 시나리오:**
    *   **Streamlit 앱:** 사용자는 앱을 실행하여 감정(예: 행복)과 상황(예: 출근길)을 선택/입력 후 '추천받기' 버튼을 누릅니다. 시스템은 다양한 소스에서 음악을 탐색하고, AI가 "밝은 댄스곡들이 출근길 활기찬 분위기를 더해주고, 경쾌한 리듬으로 하루를 시작하는 데 도움을 줍니다."와 같이 추천 이유를 설명합니다.
    *   **웹 서비스:** 사용자는 웹 브라우저로 접속하여 유사한 과정을 거쳐 음악을 추천받습니다. 웹 UI는 더욱 세련된 디자인과 로딩 프로그레스 바 등 풍부한 사용자 경험을 제공합니다.

7. 개발 흐름 및 주요 변경 사항 (실제 프로젝트 진행 과정)

본 프로젝트는 다음과 같은 단계별로 발전해왔습니다.

7.1. 1차 개발: 데이터 파이프라인 및 Streamlit MVP (`app.py` 버전)
*   **주요 목표:** 지니뮤직 차트 데이터 수집 및 분석 자동화, 그리고 감정 기반 음악 추천의 핵심 로직 구현.
*   **진행 내용:**
    *   **Airflow DAGs 구현:** `crawler_genie.py`, `diff_genie.py`, `jsontxt_genie.py`를 개발하여 데이터 크롤링, 차트 변화 분석, LLM 보고서 생성을 자동화하는 파이프라인의 기반을 마련했습니다. 이 단계에서 `data/` 폴더에 CSV, JSON, TXT 파일들이 생성되기 시작했습니다.
    *   **음악 추천 핵심 로직 (`MusicRecommender` 클래스) 개발:** Spotify, Last.fm, 지니뮤직 차트 데이터를 활용하여 음악을 검색하고, Google Gemini 등 LLM을 연동하여 추천 이유를 생성하는 백엔드 로직을 구현했습니다.
    *   **Streamlit MVP (`app.py`) 개발:** 개발된 추천 로직을 시연하고 사용자 피드백을 빠르게 얻기 위해 Streamlit을 활용한 최소 기능 제품(MVP)을 만들었습니다. 간단한 감정 선택(드롭다운), 상황 입력, 추천 결과 표시 기능을 포함했습니다. 이 버전에서는 사이드바에 연결 상태를 표시했습니다.

7.2. 2차 개선: Streamlit UI/UX 개선 (`app2.py` 버전)
*   **주요 목표:** Streamlit 앱의 사용자 인터페이스(UI) 및 사용자 경험(UX) 개선을 통한 서비스 완성도 향상.
*   **진행 내용:**
    *   **UI 레이아웃 재구성:** `app.py`에서 사용하던 사이드바 대신, 모든 콘텐츠를 중앙의 단일 컨테이너에 배치하여 시각적인 집중도를 높이고 깔끔한 디자인을 구현했습니다.
    *   **감정 선택 UI 변경:** 드롭다운 방식에서 카드형 버튼 방식으로 감정 선택 UI를 변경하여 더욱 직관적이고 시각적으로 매력적인 상호작용을 제공했습니다. 사용자 클릭 시 `st.rerun()`을 통해 UI 상태를 업데이트하도록 구현했습니다.
    *   **커스텀 CSS 적용:** Streamlit의 기본 스타일링 한계를 극복하기 위해 `st.markdown(unsafe_allow_html=True)`를 활용하여 직접 CSS를 주입, 버튼, 상태 칩 등 주요 UI 요소의 디자인을 세련되게 개선했습니다.
    *   **연결 상태 시각화:** 사이드바에 있던 연결 상태 정보를 메인 화면 하단에 3개의 컬럼으로 배치하고, CSS를 통해 상태 칩(status-chip) 형태로 구현하여 한눈에 확인할 수 있도록 했습니다.
*   **결과:** `app2.py`는 `app.py`의 핵심 기능을 유지하면서, 시각적으로 더욱 완성도 높은 Streamlit 애플리케이션으로 발전했습니다.

7.3. 3차 확장: 웹 프론트엔드 연동 및 배포 준비 (`server.py`, `index.html` 등)
*   **주요 목표:** Streamlit 앱을 넘어 일반 웹 환경에서도 서비스가 가능하도록 Flask API 서버 및 웹 프론트엔드 구축, 그리고 배포를 위한 기반 마련.
*   **진행 내용:**
    *   **Flask API 서버 (`server.py`) 개발:** Streamlit 앱의 `MusicRecommender` 클래스 로직을 재활용하여 `/api/recommend` 및 `/api/status` 엔드포인트를 제공하는 Flask 기반의 RESTful API 서버를 구축했습니다. `flask-cors`를 사용하여 CORS 문제를 해결했습니다.
    *   **웹 프론트엔드 (`index.html`, `js/main.js`, `css/main.css`) 개발:**
        *   HTML 마크업을 통해 서비스의 기본 구조와 사용자 입력 폼, 결과 표시 영역, 연결 상태 표시 영역 등을 설계했습니다.
        *   `css/main.css`를 통해 Streamlit 앱에서 개선했던 UI/UX 디자인 철학을 웹 환경에 맞게 구현, 더욱 반응적이고 현대적인 웹 디자인을 적용했습니다. 로딩 스피너, 추천 카드 등 동적인 요소들의 스타일도 포함했습니다.
        *   `js/main.js`를 통해 사용자 인터랙션을 처리하고, Flask API 서버와 통신하여 데이터를 주고받고, 받은 추천 결과를 동적으로 웹 페이지에 렌더링하는 클라이언트 사이드 로직을 구현했습니다. 로딩 프로그레스 바와 같은 시각적 피드백도 추가했습니다.
    *   **Docker 및 Docker Compose 환경 설정:** `Dockerfile`을 작성하여 Flask 애플리케이션을 컨테이너화하고, `docker-compose.yaml`을 통해 추후 Airflow 등 다른 서비스와 함께 통합 관리할 수 있는 배포 환경의 기반을 마련했습니다.
    *   **배치 파일 (`LLMUSIC.bat`):** 프로젝트 실행을 위한 간단한 윈도우 배치 파일을 구성하여 개발 및 테스트 편의성을 높였습니다.
*   **결과:** Streamlit 앱과 별개로 독립적인 웹 서비스를 제공할 수 있는 구조를 완성했으며, 향후 실제 운영 환경으로의 배포 가능성을 열었습니다.

8. 기대 효과

*   **사용자 경험 혁신:** 감정과 상황에 따른 초개인화된 음악 추천으로 사용자 만족도와 서비스 몰입도를 극대화합니다.
*   **음악 시장 인사이트 심화:** 자동화된 차트 분석 및 LLM 기반 보고서로 음악 트렌드 파악 시간을 단축하고, 데이터 기반의 전략 수립을 지원합니다.
*   **운영 효율성 증대:** Airflow를 통한 데이터 파이프라인 자동화로 수동 작업 부담을 줄이고, 자원 활용 효율성을 높입니다.
*   **AI 기술 활용 가치 증명:** LLM을 활용한 추천 설명 및 보고서 생성은 AI가 단순 정보 제공을 넘어 고차원적인 해석 및 소통 능력을 제공할 수 있음을 보여줍니다.
*   **개발 생산성 향상:** Docker 기반의 환경 구성으로 개발 및 배포 과정의 일관성과 효율성을 확보합니다.

9. 향후 확장 계획

*   **데이터베이스 통합:** 현재 파일 기반의 데이터 저장 방식(CSV, JSON)을 관계형 데이터베이스(PostgreSQL 등) 또는 NoSQL 데이터베이스(MongoDB 등)로 전환하여 데이터 관리의 효율성, 쿼리 성능, 확장성을 대폭 향상시킵니다.
*   **추천 알고리즘 고도화:**
    *   사용자 피드백 (좋아요, 스킵 등)을 반영한 강화 학습 기반의 추천 모델 도입.
    *   콘텐츠 기반 필터링(음악의 특징 분석) 및 협업 필터링(사용자 간 유사도) 결합.
    *   다양한 임베딩 기술 (Word2Vec, BERT 등)을 활용하여 곡/아티스트/감정 임베딩 공간 구축.
*   **LLM 모델 최적화 및 파인튜닝:**
    *   음악 도메인에 특화된 데이터셋으로 LLM을 파인튜닝하여 추천 설명 및 보고서 생성의 정확성과 자연스러움을 더욱 개선.
    *   경량화된 LLM 모델을 온프레미스 또는 엣지 디바이스에서 실행하여 추론 속도 향상 및 비용 절감.
*   **사용자 인증 및 개인화 기능 강화:**
    *   사용자 로그인 및 프로필 관리 시스템 구축.
    *   개인별 추천 이력, 선호 감정/장르/아티스트 데이터를 기반으로 한 장기적인 개인화 추천.
    *   플레이리스트 생성 및 공유 기능 추가.
*   **CI/CD (Continuous Integration/Continuous Deployment) 파이프라인 구축:**
    *   GitLab CI/CD, GitHub Actions 또는 Jenkins 등을 활용하여 코드 변경 시 자동으로 테스트, 빌드, 배포가 이루어지도록 시스템 구축.
    *   Docker 컨테이너 이미지 자동 빌드 및 배포.
*   **모니터링 및 로깅 시스템:**
    *   Prometheus, Grafana 등을 활용하여 시스템 성능, API 호출 상태, LLM 응답 시간 등을 실시간으로 모니터링.
    *   중앙 집중식 로깅 시스템 (ELK Stack 등)을 구축하여 문제 발생 시 신속한 진단 및 해결.
*   **확장된 데이터 소스 통합:** 멜론, 벅스 등 다른 음원 차트 데이터를 추가로 통합하여 분석 범위 확대 및 비교 분석 강화.
*   **웹소켓(WebSocket)을 이용한 실시간 기능:**
    *   실시간 차트 순위 변동 알림.
    *   다른 사용자의 추천 활동 공유.
